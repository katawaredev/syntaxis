# Pi settings and resources

Syntaxis exposes Pi management inside the AI module. The top-level sidebar has
Chat and Settings tabs. The General page groups the small sets of
agent-relevant settings into labeled subsections; prompt templates, skills, and
extensions follow it. Management work runs on the workspace server, not in the
browser.

## Extensions

Discovery queries npm's structured search API for the `pi-package` keyword,
loads each candidate's `latest` manifest, and obtains its last-month download
count from npm's downloads API. Results are requested in 20-candidate pages and
shown as a compact filterable card catalog. Name, type, installation state, and
sort controls operate on the loaded packages; "Load more" appends another page.
Installed badges are derived from the user and current project's Pi settings.
Install and uninstall buttons invoke Pi's supported CLI with argument arrays
(`pi install npm:<name>` and `pi remove npm:<name>`), never through a shell.

Current limitations:

- Pi does not publish a catalog API. Discovery depends on npm's public registry
  API and its `pi-package` convention. Git and local-only packages are not
  discoverable.
- Each "Load more" request inspects the next 20 npm candidates. The catalog
  total is npm's broader `pi-package` search count. Filtering and sorting are
  exact among the packages loaded so far;
  npm's popularity-only search ordering determines which candidates arrive on
  earlier pages.
- Package names link to npm for full metadata and README content. Syntaxis does
  not fetch or render a second package-detail document.
- The UI manages user-scoped npm packages. It recognizes project-scoped
  packages but deliberately does not remove or replace them.
- Pi extensions execute arbitrary code as the server user. Syntaxis does not
  sandbox, audit, or verify packages. The confirmation dialog is a warning, not
  a security boundary.
- Package changes are not injected into already-running RPC sessions because Pi
  exposes no host-side reload RPC. They apply to newly started chats (or a chat
  reloaded outside this UI).
- CLI operations have a three-minute timeout. Partial npm/git work left by an
  interrupted Pi command is handled by Pi on its next reconciliation.
- User-scoped packages are shared by every Syntaxis user served by the same OS
  account. Deployments must authenticate and authorize the management routes.

The UI calls Syntaxis-owned package-search and management server functions
rather than npm or the Pi CLI directly. If Pi publishes a catalog or
package-management RPC later, the server implementation can be replaced without
changing the browser-facing workflow.

## Settings schema

The editable settings list is generated by
`scripts/generate-pi-settings.mjs` from the settings documentation and
TypeScript declarations bundled with the pinned
`@earendil-works/pi-coding-agent` development dependency. A short policy list
selects settings relevant to Syntaxis; labels, descriptions, types, and defaults
come from Pi rather than being copied into the generator. The generator verifies
that every exposed field still exists, then writes
`src/ai/generated_settings.rs`. `just build-assets` refreshes this alongside the
other generated assets.

Writes use the installed Pi package's public `SettingsManager` setters and
flush mechanism. Syntaxis only enables editing when the installed Pi version
exactly matches the generated schema version and its loadable npm module is
present. Controls update optimistically while saving and return to their
previous value only when Pi reports a write error.

Current limitations:

- The generated UI is a curated agent-relevant subset of Pi settings.
  Terminal/TUI presentation and navigation settings are deliberately omitted:
  themes, external editor behavior, startup/changelog presentation, escape/tree
  controls, editor/autocomplete/cursor layout, terminal image rendering, model
  cycling, and terminal Markdown indentation. Operator-level settings are also
  omitted: session storage paths, shell/npm/proxy configuration, transports and
  network timeouts, detailed retry delays, and token-budget tuning. Resource
  arrays are managed by the corresponding resource sections.
- Standalone Pi binaries and custom wrappers may not expose the JavaScript
  `SettingsManager`. Their settings remain readable but are not editable.
- Settings are global (`~/.pi/agent/settings.json` or
  `PI_CODING_AGENT_DIR/settings.json`). Project overrides are displayed only
  indirectly through package badges and are not edited.
- Running chats retain their loaded settings and resources until Pi reloads
  them. New chats use the updated values.
- Updating Pi to a version newer than the generated schema intentionally makes
  settings read-only until Syntaxis pins that Pi version and regenerates the
  metadata. This prevents a stale UI from writing changed semantics.

## Prompt templates

The Prompt templates section manages Pi's global
`~/.pi/agent/prompts/*.md` and project `.pi/prompts/*.md` files. It supports
create, list, edit, rename, and delete operations. The editor exposes Pi's
optional `description` and `argument-hint` frontmatter plus the Markdown prompt
body. Project templates are only loaded by Pi after the project is trusted.

## Skills

The Skills section has two related workflows:

- Native CRUD for directory-based and single-file skills in
  `~/.pi/agent/skills` and `.pi/skills`. Editing a skill preserves unknown
  frontmatter and any sibling scripts, references, and assets.
- Discovery through the public `skills.sh` search and download APIs. Search,
  snapshot validation, safe-path checks, and installation are implemented in
  Rust; Syntaxis does not install or execute the `vercel-labs/skills` package.
  Search cards link to the corresponding skills.sh page. Since the catalog API
  has no offset pagination, "Load more" requests a larger result window and
  appends the new slice, up to 100 results.
- The All time, Trending, and Hot views use skills.sh's official paginated v1
  leaderboard API. That API requires `VERCEL_OIDC_TOKEN`; the view selector is
  not enabled when the server does not have one. Syntaxis does not scrape the
  skills.sh pages or provide an unofficial fallback.

Each catalog card offers separate Project and Global install buttons.
Downloaded skills are installed directly into Pi's corresponding skill
directory. Downloads are size-limited, paths must remain relative, and existing
skills are never overwritten. Skills can contain executable code and
instructions with the server user's permissions, so the UI tells users to
review installed content. Project skills are only loaded after project trust.

The skills catalog is an external convenience service rather than part of Pi's
API. If its API changes, local skill CRUD continues to work.

## AI URLs

Chat URLs identify the selected conversation with `?sessionId=...`. Settings
use the session-independent `/workspaces/:slug/ai/settings` route. Switching
back to Chat restores the selected session URL.

## Update button

"Check for updates" runs `pi update --self`. Pi performs its normal version
check and updates when appropriate; this is not a read-only availability probe.
Existing Pi processes keep running their already-loaded code, while new chats
use the updated installation. The operation result is shown in a toast.
